# Mockup → LayerPulse handover bridge

**The problem this solves.** The mockup is fast and flexible; the LayerPulse SaaS build is deliberate and rigid. We want iteration speed in the mockup, then a clean, complete hand-off so the product owner can slot the work into the LayerPulse backlog in the format their standards require — without re-deriving anything.

**The principle.** A screen isn't "handed over" as a pile of files. It's handed over as a **bundle** with four artifact types, each serving a different reader, plus a live link. The bundle is designed so the LP-side `/prd` process (or the PO by hand) can produce a canonical backlog PRD with zero guesswork.

---

## The handover bundle (per screen)

| Artifact | Path | Reader | Why it's in the bundle |
|---|---|---|---|
| **Screen narrative** | `docs/screens/<screen>.md` | Build team / LP-side `/prd` | The design source of truth — happy path, edge states, components, data joins. |
| **PRD (companion draft)** | `docs/prds/<screen>.md` | Build team / LP-side `/prd` | Problem · goals · FRs · data contract · telemetry · rollout · open questions. Feeds the canonical LP PRD. |
| **Business review page** | `public/review/<screen>.html` → `/review/<screen>.html` | **Product owner** | Plain-language, scannable, no markdown load. The PO's decision surface. Generated by the `review-page` skill. |
| **Functional screenshots** | `docs/handover/screenshots/<screen>/*.png` | Build team + LP backlog | So LP *sees* the screen, not just reads about it. The backlog item needs visuals. |
| **Live preview** | `https://layerpulze-mockup.vercel.app/<route>` | Everyone | The working, clickable mockup. |

**One link to share:** `https://layerpulze-mockup.vercel.app/review/` — the hub. It cards every screen with Review / Live / Spec links, so the PO opens one URL and can reach everything.

## The flow

```
   MOCKUP (fast · flexible)                          LAYERPULSE (rigid · deliberate)
   ────────────────────────                          ──────────────────────────────
1. Design the screen live   ─┐
2. Write screen.md + prd.md  │   ── handover bundle ──►   PO reviews via /review/<screen>.html
3. /review-page → review.html│                                      │
4. Capture screenshots      ─┘                                      ▼
                                                        PO runs /prd on LP-product repo
                                                        against screen.md + prd.md
                                                                    │
                                                                    ▼
                                                        Canonical PRD + Tier row in BACKLOG.md
                                                        (PO's format, PO's standards)
                                                                    │
                                                                    ▼
                                                        /build-feature ships the rigid SaaS build
```

The mockup designer **offers** the bundle; the PO **accepts** it into the backlog. The mockup never writes the LP backlog directly (that's the PO's domain + `/ship-it` Gate 6b autogen).

---

## The "offer" step — what the mockup does at hand-off

When a screen's design is signed off, the mockup designer should proactively assemble + offer the bundle:

1. **Confirm the four artifacts exist** (screen.md · prd.md · review.html · screenshots). Generate any that are missing — `review-page` skill for the HTML; the browser tooling for screenshots.
2. **Add/refresh the screen's card** in `public/review/index.html` (the hub).
3. **Post the handover block** in the chat reply:

   > 📦 **Handover bundle — `<screen>`**
   > 🔗 Review (PO): `/review/<screen>.html`
   > 🔗 Live: `/<route>`
   > 📄 Spec: `docs/screens/<screen>.md` · `docs/prds/<screen>.md`
   > 🖼 Screenshots: `docs/handover/screenshots/<screen>/`
   > → Ready to slot into the LayerPulse backlog. Run `/prd` on the LP-product repo against the screen + PRD when you want the canonical row.

4. **Cross-check the backlog** (per CLAUDE.md): grep `<lp-product>/docs/BACKLOG.md` for an existing Tier row or PRD covering the same area, and call out the correct pillar so the PO doesn't mis-slot it.

## Screenshots — capture convention

LayerPulse backlog items need visuals. Capture per screen:
- The **primary view** (the screen as it lands).
- Each **key interaction** (drawer open, modal open, the audience variants for Documents, etc.).

Store at `docs/handover/screenshots/<screen>/NN-<label>.png`. Capture via the Claude-in-Chrome browser tooling against the live preview, or the operator drops them in. Reference them from the review page where it helps the PO.

> Note: screenshots are the one artifact that can't be auto-generated reliably headless in this setup (the browser extension is intermittent). When it's connected, capture them as part of the offer; otherwise flag that they're pending and the operator can grab them.

---

## Why HTML for the PO surface (validated)

Per Anthropic's "unreasonable effectiveness of HTML": markdown flattens spatial information and carries high cognitive load for review. A self-contained HTML page is visual, scannable, shareable by link, and keeps the human in the loop. The `review-page` skill standardizes this so every handover bundle includes a consistent, branded PO surface. The detailed `.md` stays in the repo for the build team — two readers, two formats, one source.
