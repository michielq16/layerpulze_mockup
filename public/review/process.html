How we design a screen — start to backlog

One strict, repeatable path. Same six phases every time, so the outcome is predictable: a clean bundle that drops into the LayerPulse backlog attached to a Tier, with zero rework.

The shape

Six phases, two actors

You (the PO) own the bookends — framing the intent and accepting into the backlog. I own the middle — design, bundle, ship. The handoffs between us are explicit so nothing slips.

0 FRAME ──▶ 1 DESIGN ──▶ 2 ITERATE ──▶ 3 BUNDLE ──▶ 4 SHIP ──▶ 5 BACKLOG (you+me) (me) (you+me) (me) (you say (you accept, gates first build+narrate feedback loop 4 artifacts "merge") I draft tier)
The path

Phase by phase

0

Frame the gate

You + me · before anything is drawn
▸ You start it: "/brainstorming — I want a screen for <area>", or just name the screen.

I run the 5 fight-for-value gates on the idea before sketching a single pixel. A weak answer on any gate → I push back and counter-propose. This is the senior-PM safety valve against building generic dashboards.

Done when: we have a one-liner — pillar + persona + the decision it supports + value-loop quadrant + the DB join no other tool delivers.
1

Design

Me

I build the working screen (React/JSX) with Fabric-plausible fake data, locked brand tokens and components, and write the screen narrative (docs/screens/<x>.md). I push to a feature branch, open a PR, and link the live Vercel preview.

Done when: a live preview URL + the screen narrative exist, posted to you.
2

Iterate

You + me · the loop

You react to the live screen; I push changes; the preview updates at the same URL. This is where the mockup's speed pays off — we go back and forth until it's right. No artifact is "final" until you sign off.

Done when: you say "approved / sign off."
3

Bundle

Me · proactively, at sign-off

The moment you approve, I assemble the full handover bundle without being asked:

  • Review page — this plain-language HTML (public/review/<x>.html)
  • PRD — the build spec (docs/prds/<x>.md)
  • Screenshots — functional captures (docs/handover/screenshots/<x>/)
  • Hub card — refreshed on /review/

I also grep the LayerPulse backlog to name the correct pillar and check no duplicate row already covers it.

Done when: all five artifacts exist — screen.md · prd.md · review.html · screenshots · live route — and the hub shows the card.
4

Ship to mockup prod

You trigger · I execute
▸ You say: "merge".

I squash-merge the PR to main; Vercel redeploys production; the bundle goes live on the review hub. Then I run /sync-memory to persist the session (vault + memory + dashboard).

Done when: the bundle is live on the production hub and the session is logged.
5

Feed the backlog attach to a tier

I offer · you accept

I draft the Tier-row text for the LayerPulse backlog (status · rationale · dependencies · estimate · pillar). You paste it into <lp-product>/docs/BACKLOG.md attached to a Tier (T0–T3) — the mockup never writes the LP backlog directly. From there it's the product track: /prd formalizes → /build-feature ships → real product.

Done when: a Tier row exists in BACKLOG.md referencing the bundle. The screen has crossed the bridge.
The Phase-0 filter

The 5 gates every screen must pass

No screen reaches Phase 1 until all five have a strong answer. This is the single most important step — it's what keeps the backlog full of high-value screens instead of generic dashboards.

GateQuestionRejectProceed
G1 PillarWhich pillar does it advance?"generic dashboard""FinOps — new wasted-spend dimension"
G2 JoinWhat DB join does it need that no tool delivers?"shows the data""joins activity × licenses × tenant settings"
G3 PersonaWhose KPI does it move?"helpful for whoever""Partner QBR — cost-saving evidence"
G4 DifferentiatorWhat's unique vs. Microsoft?"nicer charts""portfolio-wide / auditor-ready"
G5 DecisionWhat decision results?"be informed""which capacity to scale down"
Your side

How to start engaging — four openers

Pick the opener that matches your intent. Each routes into the process at the right phase.

New screen (most common)

Starts at Phase 0. We frame with the gates first.

/brainstorming — I want a screen for <area>

Iterate an existing mockup screen

Jumps to Phase 2 (existing screens aren't re-gated).

iterate /<route>: <your feedback>

Bundle a screen that's already built

Jumps to Phase 3 — generate the handover artifacts.

bundle <screen> for handover

Fix a bug on a live mockup screen

Minimal path — fix, push, you confirm. No re-gate.

fix <bug> on /<route>
Why two formats

One source, two readers

Every screen produces both a markdown spec and an HTML review page from the same design. They serve different readers — neither is a summary of the other.

You (PO)the review.html — plain language, scannable, ends with decisions. Low cognitive load.
Build teamthe screen.md + prd.md — full detail, data contracts, edge states, telemetry.
LayerPulsethe screenshots — sees the screen, not just reads about it.
