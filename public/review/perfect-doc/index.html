The Perfect Document — Persona Series

One intelligence engine, four documents. Each persona reads the story shaped for them — different length, different opening, different call to action. Same underlying telemetry, same source of truth, no content drift.

The shift

From "one doc, four lenses" to "one engine, four documents"

Today's Perfect Document ships one canonical render with audience filters that hide or show sections. That works for filtering, but the personas don't actually want the same document with fewer pages — they want different documents. The BI Manager wants a 7-page briefing with a verdict; the Engineer wants 28 pages of ranked optimizations. Different attention budgets, different decisions, different narrative voices. This series demonstrates the four documents that one shared intelligence engine could generate.

Sample data scenario

Acme Manufacturing N.V. · Finance_Global_Model

All four documents are generated from the same model: a Dutch enterprise's mid-market Finance model on an F64 Premium capacity, West Europe, 48 tables / 412 measures / 18.2 GB / 12K users / €4,820/mo, owned by the Finance BI Team. The numbers and findings tell a consistent story; the four documents tell different stories about the same telemetry.

Design language

Why the docs look the way they do

Each document is built to be printable as a Word-quality deliverable — A4 page geometry, named author and date, page numbers, header/footer marks, conservative document typography (Source Serif body + Inter headings). Brand presence is tasteful: navy section bars, gold for executive callouts, status pills only where actionable. The audience receives something that reads like an audit-firm or McKinsey-style deliverable, not a SaaS dashboard.

Inline ✓ Live, ⚠ Partial, and ✗ Gap chips on every section heading make the data-fit story honest: where the prototype reads from LP-collected telemetry today, where coverage is partial, and where new collectors are needed. The companion gap audit (next section) lists each gap as a backlog-ready row.

What this proves

Same engine, four storylines, zero content drift

Every number in every document is derived from the same underlying telemetry rollup, Scanner introspection, BPA quality engine, glossary, ownership tags, capacity costs, and activity events. When the source updates, all four documents update consistently. There is no risk of the BI Manager doc claiming the refresh SLA is 99% while the FinOps doc says 98% — both read the same value, framed for their respective readers. The "different narrative voice, same source of truth" rule is structurally guaranteed.
