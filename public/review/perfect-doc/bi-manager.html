02The three risks · in detail

⚠ Mostly live

Each risk has been pre-scoped to a single owner and a single decision. The board does not need to debate the diagnosis — only approve the action and confirm the timeline. Detailed evidence behind each item lives in the appendix.

Risk 1 · Sensitivity labels missing on HR-related tables

The internal compliance posture report identified that the DimEmployee and FactCompensation tables are joined into the Finance Global Model but carry no sensitivity labels — neither in Scanner metadata nor in the Purview catalog. Both contain salary data. Under the company's ISO 27001 compliance program, undeclared sensitive data joined to a broadly-shared asset would be flagged at the next audit.

Why the board should care

The next compliance audit window opens 2026-10-18 (162 days away). Sensitivity classification is a 2-week effort if scheduled now; a 3-month panic if scheduled when the audit letter arrives. The risk is operational schedule slippage, not data exposure (RLS evaluation passes — see §8).

Risk 2 · Refresh runtime +18% after May change

The model's median refresh duration rose from 28 min to 33 min after the FactTelemetry table was added in PR #347. While the model is well within the F64 capacity headroom today, the trajectory — if unaddressed — projects a 45-minute median refresh by end-Q3 and approaches the 60-minute window beyond which capacity scaling becomes a contracted-cost discussion.

The technical lever

Enabling incremental refresh on FactTelemetry would reduce refresh time by an estimated 42% (per LayerPulse's BPA engine, rule BPA-INC-001). This is a delivery-team task — see Decision 2 below.

Risk 3 · Two reports dormant for >120 days

The Treasury Variance Weekly and Procurement Heatmap Q1 reports have not been viewed in 124 and 138 days respectively, but continue to trigger refresh and storage costs. Combined dormant cost: ~€440/month. Both were built for one-time business reviews that ended.

What is NOT a risk (worth noting)

  • Refresh failure rate (1.8%) is well within the 2% SLA threshold; the May spike was isolated to a single gateway timeout incident.
  • Model ownership is fully assigned (Maria Hendriks, Finance BI Team), unlike 3 of Acme's other 14 enterprise models. Ownership is a leading indicator of recovery time during incidents; this model is in a good position.
  • RLS evaluation passes — no role-based access leakage detected across the 41 dependent reports.

What's behind those failures

The two failed refreshes this period were both downstream gateway events — one a transient timeout (recovered on retry) and one a SQL source deadlock that was patched on the same day. Neither is a model-quality issue; both surfaced through standing-subscription alerts before any consumer-facing report was affected. The customer's gateway team has the root-cause analysis on file.

Deployment drift

Production is currently 2 releases behind DEV. The pending releases are scheduled for the next UAT-to-PRD window (June 6). No material risk; tracking only because deployment drift >3 releases is a leading indicator of incident clustering — current state is well under the threshold.
