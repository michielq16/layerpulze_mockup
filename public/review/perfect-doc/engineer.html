Architecture observations

  • Star schema discipline: 46/48 tables follow strict dimension/fact roles. Two tables (FactBudgetActual, FactForecast) are bridge tables — they break the strict star but the design is intentional and well-documented.
  • DimDate role hierarchy: single conformed DimDate referenced by 14 fact tables. The role hierarchy uses USERELATIONSHIP for fiscal-vs-calendar alternates — a clean pattern.
  • SCD2 dimensions: 8 dimensions track historical changes (DimCustomer, DimVendor, DimEmployee, DimGLAccount, etc.). Pattern is consistent: surrogate key + ValidFrom/ValidTo columns + Current_Flag.
  • Calc groups: 4 calc groups (Time Intelligence, Currency Conversion, Granularity Hierarchy, Comparison). The Currency Conversion calc group is the source of the §06 DAX optimization.

FactTelemetry — the problem table

FactTelemetry is the single largest table (8.1 GB, 142M rows) and carries the worst diagnostics. Three problems stack:

  • 61% column waste (14 of 23 columns unused). The original ingestion captured everything Scanner returned; nobody pruned. Verified via cross-reference with all 41 dependent reports + Excel Live queries — these 14 columns have zero dependencies.
  • Full-refresh strategy. No incremental partitioning. Each refresh re-reads all 142M rows from the Lakehouse and rebuilds VertiPaq compression. 44 minutes per refresh.
  • High-cardinality columns. CustomerEmail column (1.8M distinct values) accounts for 18% of the table's compressed memory and is unused in reports. This is the single biggest column-level optimization in the model.
Sprint allocation suggestion

Allocate one sprint to FactTelemetry alone: implement incremental refresh + remove the 14 unused columns + hash the high-cardinality email. Expected outcome: 8.1 GB → ~5.5 GB compressed, refresh time 44 min → ~16 min, refresh cost €680 → ~€220/mo. Risk: minimal (no consumer-facing impact); rollback is a TMDL revert.

Why bi-directional is expensive here

The three bi-directional relationships chain across FactProcurement queries: every measure that touches Procurement spend traverses the DimCurrency filter, which propagates back through FactSales (used by the same measure for context). The DAX engine evaluates this chain on every query — adding ~120ms per query × ~1,180 queries/day × 30 days = 71 minutes/month of preventable compute. Cost: ~€80/mo. Behavior: identical with single-direction (verified in sandbox).

Currency conversion — the root pattern

Three measures (Total Margin YTD Converted, Forecast Variance EUR, Procurement Spend YTD Converted) all use the same anti-pattern: CALCULATE chained inside SUMX with per-row currency lookup via the CurrencyConversion calc-group. The current implementation:

// Current implementation — Total Margin YTD Converted
[Total Margin YTD Converted] :=
VAR CurrentDate = MAX(DimDate[Date])
VAR ConvertedSales =
    SUMX(
        FILTER(FactSales, FactSales[Date] <= CurrentDate),
        FactSales[Amount] * RELATED(DimCurrency[RateToEUR])
    )
VAR ConvertedCOGS = SUMX(...similar pattern...)
RETURN
    ConvertedSales - ConvertedCOGS

Cost: per-row currency lookup × ~88M rows × per-query × ~820 queries/day. Avg P95 query latency: 18.2s.

Recommended optimization

Pre-convert in the Silver layer so the semantic model never sees source-currency amounts. Add a silver.fact_sales_converted view that materializes amount_eur via a join to dim_currency at ingest time. The measure simplifies to a plain SUM.

// Optimized implementation
[Total Margin YTD Converted] :=
CALCULATE(
    SUM(FactSales[AmountEUR]) - SUM(FactSales[COGSEUR]),
    DATESYTD(DimDate[Date])
)

// Expected: P95 query latency 18.2s → 1.8s (90% reduction)
// Cost impact: ~€120/month savings (currency-conversion chain is gone)
Rollout strategy

Pre-convert at Silver is a non-trivial change (requires Lakehouse pipeline modification + business-logic verification). Recommended: ship in a feature branch, A/B verify against current measure for one week, then cutover. Estimated effort: 3 days BI Engineering + 2 days Data Engineering coordination.
